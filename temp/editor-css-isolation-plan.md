# Editor CSS Isolation ŌĆö Next-Release Plan

## Goal

Make `@motion-proto/live-tokens` self-contained on a fresh `npm install`. A second consumer should be able to:

```ts
// consumer's main.ts
import './styles/tokens.css';        // their own theme
import { initEditorStore, /* ŌĆ” */ } from '@motion-proto/live-tokens';
```

ŌĆ”mount `<Editor />` or `<ComponentEditorPage />`, and have everything render correctly ŌĆö without remembering to import `form-controls.css`, `fonts.css`, or `fontawesome-free/css/all.min.css`, and without the editor's chrome adopting the consumer's site fonts/colors.

## Background ŌĆö what's wrong today

| File | Status | Problem |
|---|---|---|
| `ui-editor.css` | Ō£ō patched | Was loaded via `@import` in `<style>` block; broke under `css: 'injected'`. Now JS-imported from editor pages. |
| `form-controls.css` | Orphan | Editor-internal (only 4 callers in the package; consumer doesn't reference its classes) but published as a consumer subpath. Uses theme tokens (`var(--font-sans)`, `var(--surface-neutral)`, `var(--text-primary)`) so editor inputs adopt site fonts + brand colors. |
| `fonts.css` | Misplaced | A `runegoblin-site` artifact generated by `syncFontsToCss`; references local Fraunces/Manrope woff2 files. Lives in the library for legacy reasons; consumers must bring their own fonts. |
| Font Awesome | Orphan | Editor uses `<i class="fas fa-*">` icons; CSS is loaded only by the consumer's `main.ts`. Second consumer forgets ŌåÆ no-icons editor. |

Net effect: "isolation" depends on the consumer reading `CONVENTIONS.md` and getting four separate imports right. Even when they do, theme-token bleed turns editor inputs into themed inputs.

## Proposed end state

| File | Role | Treatment |
|---|---|---|
| `tokens.css` | Variable-naming contract referenced by components (`--font-sans`, `--button-primary-text`, etc.) | Library code. Stays in package + exports. Consumers override **values**, never the names. |
| `site.css` | Unscoped element typography (`h1`, `p`, ŌĆ”) ŌĆö consumed by component previews and by consumer pages | Library code. Stays in package + exports. |
| `ui-editor.css` | Editor chrome ŌĆö `--ui-*` tokens, font override, system fonts | Library code, **auto-loaded** by editor pages. No longer a required consumer import. |
| `ui-form-controls.css` (new) | Editor's form controls, `.ui-form-*` classes, `--ui-*` tokens | Library code, **auto-loaded** by editor pages. |
| `@fortawesome/fontawesome-free` | Editor's icons | **Auto-loaded** by editor pages from `dependencies`. |
| `fonts.css` + `fonts/` | Default consumer-site fonts (Fraunces + Manrope), generated by `syncFontsToCss` | Library code (starter content for consumer sites and demos). **Stays in package.** Editor never references its values. Consumers replace at will. |

### Editor font invariant

Editor chrome uses **only** the `--ui-*` font namespace, which `ui-editor.css` defines as a pure system stack:

```css
--ui-font-sans: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif;
--ui-font-mono: ui-monospace, SFMono-Regular, "SF Mono", Menlo, Consolas, "Liberation Mono", monospace;
```

No editor-namespace rule may reference `var(--font-sans)`, `var(--font-serif)`, `var(--font-display)`, or `var(--font-mono)` ŌĆö those are the consumer's theme fonts and must not bleed into chrome. This invariant is what `form-controls.css` violated; re-tokening it to `--ui-*` is what closes the leak. Enforced by lint (see ┬¦6).

Editor pages script-import everything they need; consumer only imports their own `tokens.css` + `site.css` (or the library's, if happy with defaults).

`Editor.svelte` and `ComponentEditorPage.svelte` each script-import:
```ts
import '../styles/ui-editor.css';
import '../styles/ui-form-controls.css';
import '@fortawesome/fontawesome-free/css/all.min.css';
```

`package.json#exports` drops the per-CSS subpaths for editor-internal files. Consumers cannot ŌĆö and don't need to ŌĆö import editor CSS directly.

### 4. Rework `package.json#exports`

| Old path | New path | Reason |
|---|---|---|
| `./styles/form-controls.css` | *(removed)* | Editor-internal, auto-loaded |
| `./styles/ui-editor.css` | `./styles/ui-editor.css` | Stays as a read-only subpath for token inspection; no longer required |

### 5. Audit `src/main.ts`

## Decisions (locked)

1. **Class rename:** `.form-*` ŌåÆ `.ui-form-*`.
2. **File layout:** keep `ui-form-controls.css` separate from `ui-editor.css` ŌĆö they auto-load together from the editor pages.
3. **Missing `--ui-*` tokens:** map to closest existing ŌĆö `--font-weight-light` ŌåÆ `--ui-font-weight-normal`, `--line-height-md` ŌåÆ `--ui-line-height-normal`.
4. **`tokens.css` + `site.css`:** stay in the package as library code (structural contract + preview typography). Move their export paths from `./styles/*` to `./starter/*` to communicate "default content, override at will" without splitting into a second package. Reassess a true starter package split for 0.7+ if a consumer needs it (the likely candidate is a Foundry VTT consumer that can't tolerate unscoped element rules).
5. **Version:** `0.6.0`.

## Risks

- **Breaking change for theoretical consumers using `.form-*` classes.** Mitigation: search the npm public registry / GitHub for `@motion-proto/live-tokens` usage. Today the only consumer is `runegoblin-site` and it doesn't use these classes outside live-tokens.
- **Editor's form-control look may shift slightly** since it stops adopting consumer brand colors. That's the point, but worth screenshotting before/after in PR.
- **`fontawesome-free` is a 80KB+ CSS dep loaded for every editor mount.** Acceptable for an authoring tool; flag if anyone uses the editor in a runtime context.
